### 요구사항
- 상품 상세
    - 상품 옵션 목록 추가

### 접근
옵션을 추가해야한다는 요구사항이 왔을 때 영향범위를 이해하자.
단순히 기능 추가가 아닌 새로운 추가 설계가 필요하다.

**마이그레이션**
- 기존 데이터와 옵션 추가 이후 데이터의 정합성을 지켜야한다.
- 상품은 전부 옵션이 있는것인가? 옵션이 없으면 상품을 보는가?
    - 단순하게 만드는게 좋겠다.
    - 뭐하면 뭐. 옵션 ID가 null 이면 상품. 이런 조건이 붙으면 복잡해진다.

**클라이언트**
- 상품 상세 API
    - 옵션 목록을 같이 포함시킬것인가
    - 신규 옵션 조회 API를 추가할 것인가
        - (내 의견) N+1을 고려한다면 후자가 좋음.

기획
- 옵션 있는게 좋다면
    - 데이터 보는쪽에서도 있는게 편할것이다.
    - 쿼리 잘 때에도 어려울 수 있다.
    - 백오피스 관리할 때 어려울 수 있다.
    - 등으로 적극적인 의견을 개진하자.
        - 요구사항을 줬을 때 그냥 바로 넵 하지 않아야함.

LLM 요청

**프롬프트 작성**

**프롬프트1**
상품 옵션 추가 요청
- 상품 당 옵션은 N개
- 상품 옵션에 대한 개념 클래스 정의
    - 상품 개념이 옵션 개념을 알 필요 없다. 등 정책 정의
- 상품 상세 API에 옵션 정보를 추가

**프롬프트2**
상품 옵션 개념에 필드 정보를 상품과 유사하게 구성해주고 더 필요한 필드가 있다면 추가해줘

프롬프트는 강의 특성인지 매우 간단하게 작성하시는 편
- 당연히 실무에서는 훨씬 체계적으로 구체적으로 작성해서 요청해야함.
- 요청 시에도 구현을 단계 별로 진행하도록 해야함.
- 아이디어를 요청하는것과 구현 시 자유롭게 구현을 요청하는건 더루자 않나 싶음.

**가이드라인**
- 가이드라인에도 업데이트 요청을 포함
- 작업이 완료 될 때마다 가이드라인 문서에 싱크되도록 하는건 의미있는거 같다.
- 자동화가 된다면 더 편하겠다.

**방향성**
- 방향, 의도를 잡는건 직접해야한다.
- 방향성을 어떻게 가져가는게 좋을지 ai에게 물어서라도 하자.

**리뷰**
- ProductOption에 displayOrder가 필요할까?
    - 이미 정렬이 되어서 주는데 그걸 넣을 필요가 있을까?
    - 응답을 신뢰하지 못해서 넣는다면 그 기능을 신뢰할 수 있는걸까?
    - 정상적인 응답을 줬을것이라 믿고 사용하고 정렬 순서 번호를 넣는건 지양하는게 낫겠다.
- 불필요한 정보를 response에 포함하지 않았는지?
- 옵션을 추가했을 때 다른 시스템/기능/도메인에 어떤 영향을 일으킬까를 미리 고민, 생각해야한다.
